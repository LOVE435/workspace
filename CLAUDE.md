## 开发工作流

1. **需求** — 模糊想法 → 用户故事 + 功能清单 + MVP 范围，固化为 issue
2. **设计** — 数据库 schema → API 契约 → 组件树，写清理由
3. **编码** — 先定接口再写实现，保持可测试
4. **自测** — 写代码必须同步写测试。单元测试覆盖核心逻辑，集成测试覆盖关键接口
5. **审查** — 自审一遍 diff，检查安全、性能、可维护性
6. **提交** — 小而频繁，commit message 说明动机不是复述代码
7. **部署** — CI/CD 到预发环境，冒烟测试通过后推进生产

## 命名与目录约定

- 文件名: kebab-case
- 变量/函数: camelCase
- 类/组件: PascalCase
- 常量: UPPER_SNAKE_CASE
- 目录结构按功能模块划分，不按文件类型

## 代码交付标准

- 不交付未测试的代码
- 不交付硬编码密钥/令牌的代码
- 不交付有明显安全漏洞的代码
- 每个 PR 附带简短的变更说明

## 沟通规则

- 主动推进，不等待指令
- 带方案沟通：问题 + 选项 A/B/C + 推荐 + 请求确认
- 关键决策记录简要理由
- 错误立即承认，附带事后分析
